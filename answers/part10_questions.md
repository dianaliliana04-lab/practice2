## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C 
```

**Объяснение:**
Порядок выполнения:

Статический блок static { ... } выполняется один раз при загрузке класса, до вызова main. Выводится A.

В main сначала выводится D.

При new Demo():

Блок экземпляра { ... } выполняется перед телом конструктора. Выводится B.

Затем конструктор выводит C.

Выводится E.

Второй new Demo() снова выполняет блок экземпляра и конструктор: B C.
Итог: A D B C E B C.

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**
В подклассе Child (пакет b) доступны:

public члены Parent

protected члены Parent (так как наследование даёт доступ)

члены с пакетным доступом (default) недоступны, потому что пакеты разные.

private члены недоступны.

В другом классе пакета b, который не наследует Parent:

Только public члены Parent доступны.

protected и пакетные члены недоступны (так как нет наследования и пакеты разные).

private недоступны.

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:**
Да, можно. Если все наследники sealed-класса — record, то иерархия становится полностью исчерпывающей и неизменяемой.

Преимущество в switch: компилятор точно знает все возможные подтипы, поэтому switch по sealed-типу не требует ветки default и проверяет полноту покрытия. С record можно использовать деструктуризацию в case, получая компактный и безопасный код.

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**
a == b → false: a указывает на объект в String Pool, b — новый объект в куче.

a == c → true: c = b.intern() возвращает ссылку на строку из пула (ту же, что и a).

b == c → false: b — объект в куче, c — объект в пуле, разные адреса.

String Pool — область в куче, где JVM кэширует уникальные строковые литералы. Метод intern() проверяет, есть ли уже такая строка в пуле: если есть — возвращает ссылку на неё, иначе добавляет и возвращает ссылку.

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:**
класс C должен переопределить метод hello(), явно указав, какую версию вызывать или предоставить свою реализацию.
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello();  // или B.super.hello() или своя логика
    }
}

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**
StringBuilder — не потокобезопасный, работает быстрее. Используется в однопоточной среде.

StringBuffer — потокобезопасный (все методы синхронизированы). Используется, когда несколько потоков одновременно изменяют одну строку.

Сценарий для StringBuffer: логирование в многопоточном сервере, где несколько потоков дописывают сообщения в общий буфер перед записью в файл.

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
Если раскомментировать x = 20;, код не скомпилируется с ошибкой: local variables referenced from a lambda expression must be final or effectively final
```

**Объяснение:**
Лямбда-выражение захватывает локальную переменную x, которая должна быть effectively final (не изменяться после инициализации). Попытка изменить x после создания лямбды нарушает это требование. Компилятор запрещает такое изменение, чтобы избежать проблем с состоянием при отложенном выполнении лямбды.

## Вопрос 8

Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ваш пример и объяснение:**
Predicate<Integer> isEven = n -> n % 2 == 0;
Это лямбда-выражение нельзя заменить ссылкой на метод, потому что оно содержит логическую операцию % 2 == 0, для которой нет подходящего готового метода в Java. Ссылка на метод требует, чтобы существовал метод с сигнатурой, точно соответствующей функциональному интерфейсу (например, Integer::bitCount). Произвольные выражения заменить ссылкой на метод невозможно.

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**
EnumMap хранит значения в массиве, индексированном по ordinal() ключей-enum, обеспечивая O(1) доступ без хэширования и коллизий.

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**
Компилятор автоматически генерирует:

Канонический конструктор Point(int x, int y)

Геттеры x() и y() (без префикса get)

Методы equals(), hashCode(), toString()

Добавить изменяемое поле в record нельзя, так как все поля неявно final. Можно добавить статические поля, но не экземплярные изменяемые.

Наследоваться от record нельзя: все record неявно final, поэтому запрещено создавать подклассы.
